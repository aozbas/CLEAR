# supabase/

Supabase project config, SQL migrations, and seed data — managed by the [Supabase CLI](https://supabase.com/docs/guides/cli).

## Layout
```
supabase/
├── README.md
├── config.toml          # generated by `supabase init`
├── migrations/          # timestamped SQL migration files
│   ├── 0001_init.sql
│   ├── 0002_profile_trigger_and_constraints.sql
│   └── 0003_lock_handle_new_user.sql
└── seed.sql             # local-dev seed data
```

## Common commands
```bash
supabase init            # one-time, creates config.toml
supabase start           # local Postgres + Studio + Auth
supabase db reset        # re-apply all migrations + seed
supabase migration new <name>
supabase db push         # push migrations to the linked remote project
```

## Conventions
- Migrations are append-only. Never edit a migration that's been pushed — write a new one.
- Keep `seed.sql` to development-only fixtures. No real user data.
- Schema summary lives in [../docs/db-schema.md](../docs/db-schema.md).
